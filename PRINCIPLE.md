# git-cartographer System Principle

## Problem: 解決する課題

**現状:**
AIコーディングエージェント（Claude Code）はタスクのたびにコードベース調査を行い、トークンを大量消費する。
調査結果は揮発し、次のタスクで同じ調査が繰り返される。

**目指す状態:**
一度構築したら能動的に触らずとも、コードベースの地図が更新され続け、
Claude Codeへのコンテキスト注入コストが下がる状態。
「エージェントを賢くする」のではなく、**エージェントが泳ぐ海を賢くする**。

**設計の核心:**
LLMを一切使わない純粋な測量機械。git churn × AST依存解析（静的測量）と
CCの踏破軌跡（動的測量）を統合し、地図が自律的に重要地点を学習する。
**地図製作者は現地を解釈しない。測量して記録するだけ。**

---

## Design Principles: 設計原則

### 1. 忘れられる構造（Forgettable by Design）
- **説明**: 一度セットアップしたら、能動的なメンテナンスなしに動き続けること。
  Hooks一本化・JSON canonical 出力（Markdown は `--markdown` フラグで opt-in）・差分駆動更新により「触らなくていい状態」を実現する。
- **出力方針**: JSON（`stable.json`, `co-change.jsonl`, `hotspot.json`）が canonical データとして常時生成される。
  Markdown（`stable.md`, `co-change.md`, `hotspot.md`）は `--markdown` フラグ指定時のみ生成される閲覧用フォーマット。
- **理由**: 管理コストが高いツールは使われなくなる。
  開発者の認知負荷を増やさず、環境の一部として溶け込むことが、継続的な価値の条件。

### 2. 測量に徹する（Surveyor, Not Interpreter）
- **説明**: git-cartographerはコードベースを測量して記録するだけであり、意味の解釈・判断・提案を行わない。
  測量は以下のカテゴリに分かれる：
  - **静的測量**
    - git churn ベース: stable / hotspot / co-change（時系列統計）
    - AST / 構造ベース: components（構造的事実 — `scan_dirs` 配下の境界一覧）
  - **動的測量**: 踏破録（CC アクセスパターン）
- **理由**: 測量と解釈を分離することで、LLMゼロの軽量設計が成立する。
  構造的事実（境界）と統計的事実（churn）と痕跡（踏破録）を独立して扱うことで、
  各層の更新コスト・寿命・再現性が独立する。
  Distributed Cognition的に「知性を地図という環境に宿す」設計の純度を保つ。

### 3. 非対称更新コスト（Asymmetric Update Cost）
- **説明**: 地図を3層（stable / structure / hotspots）に分離し、
  変化頻度に応じて更新コストを非対称化する。変わらない層は触らない。
- **理由**: 地図全体を毎回再生成するコストは現実的でない。
  層ごとに更新粒度を変えることで、差分駆動モードが成立し、継続運用が可能になる。

### 4. Unix哲学の軽量スタック（Lightweight by Default）
- **説明**: gitコマンド・TreeSitter・Markdownのみで完結する。
  外部依存を増やす場合は、既存の代替手段では解決できない理由を示す。
- **理由**: 重量スタック（embedding・BM25・専用DB等）は導入・保守コストが高く、
  「忘れられる構造」と矛盾する。Worse is Better——シンプルな実装が生き残る。
- **AST実装（2026-04-12 決定）**: **TreeSitter** を採用。
  Python stdlib `ast` より外部依存は増えるが、多言語対応・部分解析・エラー耐性の恩恵が
  「stdlib ast で作り直す」後払いコストを上回ると判断。

### 5. Hooks一本化（Single Hook Layer）
- **説明**: 踏破録の記録とコンテキスト注入は Claude Code Hooks（PostToolUse / PreToolUse）に一本化する。
  同じ目的のプロセスを複数立ち上げない。
- **理由**: 観測レイヤーを分散させることは怠惰要件（維持コストの最小化）に反する。
  接合点が減り、故障点も減る。

## Conflict Resolution: 原則間の衝突ルール

**優先順位: セキュリティ ＞ 怠惰（忘れられる構造）＞ 測量精度 ＞ 汎用性**

| 衝突パターン | 解決ルール |
|---|---|
| 測量精度 vs 軽量スタック | 軽量スタックを優先。精度向上は、精度不足が実際に問題になってから対処する |
| 構文測量の精度向上 vs 軽量スタック | 軽量スタックを優先（tree-sitter で表現できる範囲に留める。専用パーサ追加は却下） |
| 測量データの粒度 vs 単一 canonical の維持 | 単一 canonical を優先（粒度調整・集約は agent 側で行う） |
| 汎用性 vs 怠惰 | 怠惰を優先。Claude Code固定・Markdown固定などの割り切りは明示的な設計判断として許容 |
| 新機能追加 vs 忘れられる構造 | 「触らなくていい状態を壊すか」を基準に判断。壊すなら却下 |
| セキュリティ vs 怠惰 | セキュリティが優先。踏破録はアクセスパターンのリーク資産であり、Trust Boundaryを後回しにしない |
| 解釈・判断 vs 測量 | 測量に徹する。解釈が必要な機能は別でやる |

## Non-Goals: 意図的にやらないこと

- **LLMによる意味的逸脱検知**: git-cartographerはLLMを持たない
- **コードの自動修正・リファクタリング提案**: 測量に徹する。修正判断はClaude Codeと人間に委ねる
- **CI/CDパイプラインへの組み込み**: 初期スコープ外。継続動作環境（開発ループ内）に集中する
- **複数リポジトリの横断解析**: 複雑性が非線形に増大し、軽量スタック原則と衝突する
- **UIの提供**: `output/` 配下の Markdown / JSON / JSONL が canonical 出力。HTML やインタラクティブビューアは別の関心事であり、YAGNIの典型。JSON は Markdown の派生ではなく canonical データ（Markdown は JSON から派生する閲覧用フォーマット）
- **Claude Code以外のエージェントへの対応**: Claude Code固定は怠惰判断。他エージェントへの汎用化は「忘れられる構造」の破壊につながる
- **Cynefinの自動分類実装**: 3層命名の概念語彙としてのみ使用。アルゴリズム化は設計の複雑性を上げるだけで価値に見合わない
- **Component Card の意味判断（type / edges 推定）**: components 軸では「scan_dirs 配下の直下サブディレクトリ一覧」のみを記録する。コンポーネントの種別判定・依存推定・Card 形成は agent 側責務（PRINCIPLE 2 整合）
- **コンポーネント分割の射影**: components 軸は単一 canonical（`output/components.json`）のみを出力する。粒度別の分割表現（細粒度 / 粗粒度の同時提供）は持たない

## Decisions: 設計決定済み事項

### Trust Boundary設計（2026-04-12 決定）

踏破録のアクセスパターンは**リポジトリ内に配置**（git 管理・チーム共有）。

| 論点 | 決定 |
|------|------|
| 誰が読めるか | リポジトリ内（git 管理）。チーム共有を許容 |
| 保持範囲 | コミット数ベースで半減・消滅 |
| 消去ルール | 自動消去（忘却関数によりエントリが自動的に消える） |

**補足**: アクセスパターンを git 履歴に残したくない場合は `.gitignore` で除外する。除外判断は利用者に委ねる。

### ウィンドウスキャンとスキップ最適化（2026-04-26 更新）

**常に最新 `window` コミットをスキャンする**方式を採用。`.cartographer_state` は HEAD 未変更時のスキップ最適化としてのみ使用する。

| 状態 | 動作 |
|------|------|
| `.cartographer_state` が HEAD と一致 | 実行スキップ（何もしない） |
| `.cartographer_state` が HEAD と不一致または不在 | 最新 `window` コミットをフルスキャン → ファイル生成 → 状態更新 |

**廃止した設計**: 差分駆動（incremental）モード。
理由: `window`（最新N件の地図）と incremental（delta）は時間軸が矛盾し、
コミットを重ねるほど hotspot・co-change の正確性が失われた。

**補足**: リセットしたい場合はファイルを削除すれば次回フルスキャンが走る（動作は変わらない）。

### 踏破録の忘却パラメータ（2026-04-12 決定）

指数減衰モデルを採用。式：`score(t) = initial_score × (0.5 ^ (commits_elapsed / half_life))`

| パラメータ | 値 | 備考 |
|---|---|---|
| 初期スコア | 1.0 | アクセス時に付与 |
| 半減期 | **20コミット** | 20コミット経過でスコアが半分に |
| 消滅閾値 | **0.1** | スコアがこれを下回ったらエントリ削除（約66コミットで消滅） |

具体値は実測から調整する（仮置き値）。

### co-change の effective_weight 半減期（2026-04-26 決定）

co-change の `effective_weight` 減衰には半減期 90 コミット（`halflife_commits` デフォルト）を使う。
踏破録の忘却関数（半減期 20 コミット）とは独立した値で、前者は構造的結合の劣化、後者はアクセス痕跡の忘却を表す。

計算式: `effective_weight = count × 0.5^(commits_since_last_cochange / halflife_commits)`

| 用途 | 対象 | 半減期 |
|------|------|--------|
| co-change effective_weight | 構造的結合の劣化（ペアが共変しなくなるまで） | **90コミット** |
| 踏破録スコア | アクセス痕跡の忘却（ファイルが触られなくなるまで） | **20コミット** |

2つの半減期は別の現象を表すため、独立して調整する。co-change の 90 は「3ヶ月程度の開発サイクルで構造的結合が自然に劣化する」という想定に基づく仮置き値。
